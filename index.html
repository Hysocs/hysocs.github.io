CobbleSpawnRegions

Create named regions that control Cobblemon natural spawns and custom Pokemon spawns. Use it to create custom spawns for separate towns, routes, caves, arenas, and event areas without needing to modify the natural spawn behavior per world.

Quick Start

Region Setup

  1. Run /csr giveclaimstick coords or /csr giveclaimstick chunk.
  2. Left-click a block with the stick to set point 1.
  3. Right-click a block with the stick to set point 2.
  4. Run /csr create <name> to save the selected region.
  5. Run /csr editgui or /csr edit <regionId> to configure natural and custom spawn behavior.

Claiming Demo

Region Features

Region Claims

  • Claim exact areas with a selection stick, using either block-by-block bounds or full chunk-based regions.
  • Layer multiple zones with priority, so towns, routes, caves, arenas, and event areas can overlap cleanly.
  • Preview region borders with particles before committing spawn rules, which makes setup easier for staff.
  • Manage regions in-game through commands and GUI screens instead of editing every file by hand.
  • Clone and rename regions to quickly reuse a working setup across similar areas.

Natural Spawn Rules

  • Block natural spawns entirely in protected or custom encounter areas.
  • Filter specific Pokemon by species, labels, or scanned conditions without disabling the whole ecosystem.
  • Inspect nearby Pokemon to discover useful condition values for spawn rules.
  • Use priority regions so the most specific area controls the final spawn behavior.
  • Reload and test quickly with in-game checks that show which region controls the player location.

Custom Pokemon Entries

  • Spawn your own Pokemon groups inside a region, separate from Cobblemon's normal wild spawn table.
  • Set region spawn timers so an area attempts custom spawns on your schedule.
  • Choose how many spawn attempts happen each timer cycle and cap the total active custom Pokemon.
  • Require nearby players so custom encounters only run when someone is close enough to use the area.
  • Force spawns for testing or events without waiting for the normal timer cycle.

Spawn Fields

Encounter TablesAdd region-owned Pokemon with weighted or independent spawn chances.
Level RangesSet min and max levels per Pokemon so each zone has the right difficulty.
Forms & AspectsUse forms, variants, and aspects such as shiny when building special encounters.
Habitat RulesLimit spawns by block type, time of day, and weather.
Catch ControlMake encounters uncatchable or require specific Poke Balls.
Battle TuningCustomize IV ranges, EV rewards, size, held items, and starting moves.
Population CapsKeep custom spawns from flooding an area, even across chunk unloads.
Stay In RegionKeep custom-spawned Pokemon wandering back toward their assigned zone.

Example: Spawn Town

For a protected spawn town, claim the full town as one high-priority region, set natural spawns to blocked, and leave custom spawns empty. Use /csr check at the town edge to confirm the correct region wins before reloading for players.

Common Mistakes

  • Use chunk mode for large towns or routes. Exact coordinate mode is better for small buildings, caves, arenas, and event rooms.
  • If two regions overlap, the highest priority region controls the final spawn behavior.
  • After hand-editing JSONC files, run /csr reload and then /csr check in-game.

Config Guide

JSON Rules For This Mod

  • Region files live in config/cobblespawnregions/regions.
  • Each region is one region_name.jsonc file.
  • Pokemon go inside the selectedPokemon array.
  • Objects use { }. Lists use [ ]. Text must use quotes.
  • Put a comma between fields and between Pokemon entries, but not after the last item.
  • Run /csr reload after saving changes.

Common Values

  • mode: COORDS for exact block positions or CHUNK for chunk selections.
  • spawnTime: ALL, DAY, or NIGHT.
  • spawnWeather: ALL, CLEAR, RAIN, or THUNDER.
  • spawnChanceType: COMPETITIVE joins the weighted pool; INDEPENDENT rolls its own percent first.
  • allowedBlocks: empty list for any floor, #solid, #water, #air, or exact block IDs such as minecraft:grass_block.
  • returnTarget: RANDOM, CENTER, or CLOSEST.
  • aspects: Cobblemon aspect strings for that Pokemon or form, such as shiny. Use the GUI to see valid aspects for a species.
  • formName: Cobblemon form names such as Normal or galarian; blank, normal, default, and standard are treated as the base form.

Region Field Options

  • selectedPokemon: list of Pokemon spawn objects. Empty means the region has no custom spawns.
  • requiredPokeBalls: Cobblemon ball IDs without the namespace in this config, for example poke_ball, great_ball, or ultra_ball.
  • itemsWithChance: item ID to percent chance, for example "minecraft:diamond": 5.0.
  • selectedMoves: move IDs such as tackle or quickattack; forced makes the move start on the Pokemon.
  • disallowedSpecies: species IDs such as cobblemon:pikachu.
  • disallowedLabels: Cobblemon labels such as legendary, mythical, or other labels present on installed Pokemon data.
  • exclusionConditions: condition strings gathered by /csr inspectnearest or the condition scanner GUI.

Quick Start

Reward Setup

  1. Open config/cobblebattlerewards/config.jsonc.
  2. Edit the trigger you want to reward: battleWonRewards, battleLostRewards, battleForfeitRewards, or captureRewards.
  3. Add a named reward entry such as "money_100": { ... }.
  4. Set the reward type to command or item, then fill in the command or item data.
  5. Run /cbr reload, then win, lose, forfeit, or capture in-game to test that trigger.

Reward Features

Battle Outcomes

  • Pay players for wins in wild, NPC, or PvP battles.
  • Reward captures when players catch Pokemon through battle.
  • Use loss and forfeit rewards for pity prizes, penalties, or server-specific event rules.
  • Run economy or server commands for money, crates, ranks, points, or any plugin command your server supports.
  • Give item rewards directly and choose whether full inventories drop rewards or skip them.

Reward Conditions

  • Make Pokemon-specific rewards for species, types, shinies, forms, and other Cobblemon properties.
  • Restrict by level so high-value rewards only come from tougher opponents.
  • Restrict by battle type to separate wild grinding, NPC trainer rewards, and PvP incentives.
  • Limit by dimension for location-based rewards such as Nether, End, or event-world bonuses.
  • Use blacklists to exclude common Pokemon or keep special rewards from stacking with normal ones.

Cooldowns And Messages

Chance RollsMake rewards guaranteed, rare, or anywhere in between.
CooldownsLimit each reward per player to prevent farming abuse.
Reward OrderLet rare rewards run first and block lower-value fallback rewards.
Custom MessagesShow polished MiniMessage feedback when rewards trigger.
PlaceholdersUse player, Pokemon, level, battle type, chance, coordinates, and cooldown time.
Condition ListsUse simple matches or grouped requirements for more specific rewards.

Example: First Economy Reward

Start with one low-value wild battle reward, reload with /cbr reload, and test it against a common Pokemon before adding rare species, PvP rules, or cooldown logic.

Testing Notes

  • Put rare rewards before fallback rewards when reward order matters.
  • Use cooldowns on money and crate commands before opening the config to all battle types.
  • If a reward does not trigger, check the battle type and Pokemon condition list first.

Config Guide

JSON Rules For This Mod

  • All rewards are edited in config/cobblebattlerewards/config.jsonc.
  • Reward maps are objects: "reward_name": { ... }.
  • Every reward name in the same map must be unique.
  • Text must use quotes. Lists use [ ]. Objects use { }.
  • Use commas between rewards and fields, but not after the last item.
  • Run /cbr reload after saving changes.

Common Values

  • inventoryFullBehavior: drop drops item rewards if inventory insert fails; skip does not give the item.
  • Trigger maps: battleWonRewards, battleLostRewards, battleForfeitRewards, and captureRewards.
  • type: command or item.
  • battleTypes: wild, npc, and pvp.
  • allowedDimensions: empty list for all dimensions, or IDs such as minecraft:overworld, minecraft:the_nether, minecraft:the_end, or modid:dimension_name.
  • chance: percent from 0.0 to 100.0. 100.0 is guaranteed.

Reward Field Options

  • command: server command text with no leading slash. Used when type is command.
  • itemStack: escaped ItemStack JSON string, for example {"id":"minecraft:diamond","count":1} escaped inside quotes.
  • cooldown: seconds per player for that reward. 0 means no cooldown.
  • minLevel and maxLevel: opponent Pokemon level range.
  • order: lower numbers run first. Default is 999.
  • excludedRewards: reward names from the same trigger map that should be blocked after this reward succeeds.
  • conditionsBlacklist: false means reward only matching Pokemon; true means reward anything that does not match.

Condition Values

  • Raw species match: cobblemon:pikachu, cobblemon:zapdos, or another species ID.
  • Property match: key=value or key:value, for example type=electric, type=ghost, shiny=true, or form=galarian.
  • Top-level list items are OR checks: ["type=electric", "type=flying"] matches either.
  • Nested lists are AND groups: [["type=electric", "shiny=true"]] requires both.
  • Run /cbr listconditions with a Pokemon in party slot 1 to print the condition keys the mod can read from that Pokemon.

Placeholders

  • %player%: player name.
  • %pokemon%: opponent Pokemon species.
  • %level%: opponent Pokemon level.
  • %battleType%: wild, npc, or pvp.
  • %chance%: configured reward chance.
  • %coords%: player X,Y,Z.
  • %trigger%: reward trigger.
  • %dimension%: player dimension.
  • %time%: cooldown time left, only for cooldown messages.

What Not To Use

  • Do not use reward type names other than command or item.
  • Do not include a leading slash in command.
  • Do not put plain item IDs in itemStack; it must be an escaped ItemStack JSON string.
  • Do not use battle types outside wild, npc, or pvp.
  • Do not duplicate reward names inside the same reward map.

Quick Start

Hunt Setup

  1. Run /hunts editor to set up hunts in-game, or edit the JSON files in config/cobblehunts.
  2. Add Pokemon to the hunt pools and add rewards to the loot pools.
  3. Players run /hunts to open the hunt GUI and start a global or solo hunt.
  4. Players catch a matching Pokemon, then turn it in from the hunt GUI to claim rewards.

Hunts GUI

Hunt Features

Player Turn-Ins

  • Global hunts give the whole server shared Pokemon targets and a shared countdown.
  • Solo hunts give each player personal missions with easy, normal, medium, and hard tiers.
  • Players use one GUI to view active hunts, start solo missions, check rewards, and turn in catches.
  • Matching catches are detected and can notify players when a caught Pokemon fits an active hunt.
  • Turn-ins can consume the Pokemon so servers can choose between trophy-style or spend-the-catch gameplay.

Global And Solo Hunts

  • Run multiple global hunts at the same time to create active server-wide goals.
  • Use timers and cooldowns to pace how often hunts appear and how long players have to finish them.
  • Gate hunt tiers with permissions for ranks, progression, VIP access, or event groups.
  • Track points on a leaderboard so frequent hunters have a reason to compete.
  • Auto-start solo hunts for a faster player flow, or make players choose missions manually.

Example: Weekend Hunt

For an event hunt, create a short global timer, keep the target pool small, and use command rewards for keys, points, or cosmetics. Test turn-in behavior with one caught Pokemon before adding consume rules.

  • Use permissions to hide harder solo tiers until players should see them.
  • Document whether turn-ins consume Pokemon so players do not lose trophies unexpectedly.

Config Guide

Config Files

  • config/cobblehunts/pokemon_pools.jsonc controls which Pokemon can appear in each hunt pool.

  • Run /hunts reload after saving changes.

Permission Nodes

  • cobblehunts.hunts: open the hunt GUI.
  • cobblehunts.global: access global hunts.
  • cobblehunts.solo.easy, .normal, .medium, .hard: access solo tiers.
  • cobblehunts.givebrush: give tracking brushes.
  • Default permission and op levels are both 2.

Common Values

  • rewardMode: weight picks one reward by weight; percentage rolls every reward independently.
  • Pokemon pool names: globalPokemon, soloEasyPokemon, soloNormalPokemon, soloMediumPokemon, and soloHardPokemon.
  • Loot pool names: globalLoot, soloEasyLoot, soloNormalLoot, soloMediumLoot, and soloHardLoot.
  • Loot reward type: command or item.
  • gender: null, random, male, or female. The editor cycles Any, Male, Female.
  • nature: null, random, or one of the 25 Pokemon natures.
  • ivRange: null, "1", "2", or "3"; it means at least that many IVs are 20 or higher.
  • aspects: Cobblemon aspect strings for the species/form, commonly shiny.

Nature Values

  • Adamant, Bashful, Bold, Brave, Calm, Careful, Docile, Gentle, Hardy, Hasty.
  • Impish, Jolly, Lax, Lonely, Mild, Modest, Naive, Naughty, Quiet, Quirky.
  • Rash, Relaxed, Sassy, Serious, Timid.

Teleport Features

Location Search

  • Random RTP sends players to a safe location inside your configured radius.
  • Biome RTP lets players search for a specific biome instead of teleporting blindly.
  • Dimension-aware safety supports surface, cave, floating, and automatic location checks.
  • Dimension redirects can send Nether or End requests back to the Overworld, or to another target world.
  • Busy-search protection prevents too many RTP searches from running at once.

Arrival Behavior

  • Glide landings can drop players above the destination and let them fly down instead of appearing instantly.
  • Temporary glider gear can equip a configured item for the glide and restore the player afterward.
  • Particle trails make RTP arrivals feel more polished and visible.
  • Slow loading prepares nearby chunks before releasing the player at the destination.
  • Clear feedback messages show cooldowns, biome scanning, loading progress, success, and failure states.

Permissions And Worlds

Bypass PermissionLet staff or trusted groups skip cooldowns entirely.

Example: New Player RTP

Give default players a conservative radius and cooldown, then add wider radius permission tiers for trusted groups. Test each dimension separately because surface, cave, and floating safety checks behave differently.

Operational Notes

  • Keep busy-search limits low on smaller servers to avoid many expensive biome searches at once.
  • Use slow loading when players report arriving before nearby chunks are ready.
  • Redirect Nether or End RTP only if your server rules expect players to land in another world.

Config Guide

JSON Rules For This Mod

  • Main config lives at config/blanketrtp/config.jsonc.
  • Dimension IDs use full IDs such as minecraft:overworld.
  • Biome IDs use full IDs such as minecraft:plains.
  • Cooldown durations, slow load times, and biome tick limits are numbers.
  • Run /rtp reload after saving changes.

Placeholders

  • {time}: cooldown seconds left.
  • {biome}: requested biome ID.
  • {x}, {y}, {z}: destination coordinates.
  • {percent}: slow-load progress percentage.

Common Values

  • dimensionSafetyRules.mode: auto, surface, overworld, cave, nether, floating, or end. overworld aliases surface, nether aliases cave, and end aliases floating.
  • dimension, from, and to: full dimension IDs such as minecraft:overworld, or * for wildcard safety rules.
  • gliderSlot: equipment slot name such as CHEST for an elytra-style glider item.
  • particleType: particle ID such as minecraft:cloud.
  • cooldowns.permission: permission node for that tier, or empty string for the fallback tier.
